## Graph compilation & execution mode

### `enforce_eager`

- **vLLM key**: `model_config.enforce_eager`
- **Default**: `False`
- **When `True` (eager) → profiling fingerprint**:
  - `step_anatomy.head_wall_ms / step_summary.wall_ms > 10%`
  - Many small kernels with high `wait_us`, gaps between sequential ops
- **When `False` (graph) → profiling fingerprint**:
  - `head_wall_ms / wall_ms < 5%`
  - Large fused kernel blocks, no inter-kernel gaps
- **Interaction**: vLLM-Ascend graph mode uses `enable_npugraph_ex=True` (default) for compilation. Without it, graph mode still runs but uses the torchair fallback path instead of npugraph_ex.
- **Agent note**: if `config_signatures.graph_mode = "eager_mode"`, ask whether `enforce_eager` was explicitly set to True. The default is False — eager mode in production is unusual.

### `enable_npugraph_ex`

- **vLLM-Ascend key**: `ascend_compilation_config.enable_npugraph_ex` (in `additional_config`)
- **Default**: `True`
- **Effect**: Uses the npugraph_ex compiler backend for ACL graph generation. When False, falls back to fusion_pass_compile via torchair.
- **Profiling fingerprint**: Not directly distinguishable from profiling data. Both paths produce fused ACL graphs.
- **Agent note**: only relevant for graph-mode troubleshooting. If the user reports graph capture issues, ask whether they are using npugraph_ex or torchair.

### `enable_static_kernel`

- **vLLM-Ascend key**: `ascend_compilation_config.enable_static_kernel`
- **Default**: `False`
- **Requires**: `enable_npugraph_ex=True`
- **Effect**: Pre-compiles operator binaries for specific shapes at graph capture time. Higher first-run latency, lower steady-state latency.
- **Profiling fingerprint**: Very low head_ms, aggressive fusion. Not distinguishable from standard npugraph_ex in the profile.

### `cudagraph_capture_sizes`

- **vLLM key**: `compilation_config.cudagraph_capture_sizes`
- **Default**: vLLM dynamic default (computed from max_num_seqs)
- **Profiling fingerprint when incomplete**:
  - Mixed pattern: some steps graph-like (head ~0), others eager-like (head > 10%)
  - `config_signatures.graph_mode = "partial_capture"`
- **Common cause in vLLM-Ascend**: speculative decode increases the uniform decode query length to `num_spec_tokens + 1`. If `cudagraph_capture_sizes` doesn't cover this value, decode falls back to eager.
- **Agent note**: if `partial_capture` detected, ask for `cudagraph_capture_sizes` and `num_speculative_tokens`. The capture list must cover `num_spec_tokens + 1` for decode to be fully captured.

## Compilation fusion passes

These are `AscendCompilationConfig` options (passed via `additional_config.ascend_compilation_config`). They control operator-level fusions inside the compiled graph.

### `fuse_qknorm_rope`

- **Default**: `True`
- **When True → profiling fingerprint**: No separate RoPE kernel appears in the attention block. Q/K norm and RoPE are emitted as a single fused kernel.
- **When False → profiling fingerprint**: An explicit `attention.rope` kernel appears separately after the attention norm.
- **Agent note**: if `characterizations.json` event categories include `attention.rope`, this pass is likely disabled. The default is True so check if the user explicitly disabled it.

### `fuse_allreduce_rms`

- **Default**: `False`
- **When True → profiling fingerprint**: Allreduce collective and rmsnorm appear as a single operation with no gap between them. Allreduce wall time appears shorter (no separate post-processing).
- **When False → profiling fingerprint**: Allreduce completes first, then a separate rmsnorm kernel executes. A visible gap exists between the two.
- **Agent note**: this is off by default but can significantly reduce TP overhead. If `hccl_op_summary.csv` shows heavy allreduce time in TP blocks, ask if the user has enabled this pass. Only works in graph mode with npugraph_ex.

### `fuse_norm_quant`

- **Default**: `True`
- **When True → profiling fingerprint**: Norm and quant operations are fused into a single kernel.
- **Profiling visibility**: Cannot reliably confirm from kernel names alone — the fused kernel may share the norm kernel's name. Ask the user to check their config if quant-related overhead is suspected.

## Attention backends

vLLM-Ascend selects the attention backend automatically based on model architecture and quant config. The detected backend appears in `config_signatures.attention_backend`.

### FIA (dense models)

- **Detected when**: `FusedInferAttentionScore[V2-V5]` is the dominant attention kernel
- **Used for**: Qwen, Llama, Mistral, GLM, and other dense models
- **Architecture resolution**: `gqa_or_mha` — may refine to `mha`/`gqa`/`mqa` via shape-based heuristic in `attention_families.yaml`
- **Source**: `vllm_ascend/attention/attention_v1.py` `AscendAttentionBackend`

### SFA (sparse attention)

- **Detected when**: `QuantLightningIndexer`, `KVQuantSparseAttnSharedKV`, `Compressor` kernels appear
- **Used for**: DeepSeek V3.2 (DSA), DeepSeek V4 (CSA, HCA)
- **Sub-type resolution**: Determined by kernel signature combination (see `attention_families.yaml`)
- **Source**: `vllm_ascend/attention/sfa_v1.py` `AscendSFABackend`

### MLA (multi-head latent attention)

- **Detected when**: `MlaPreprocess`/`MlaProlog`/`KvRmsNormRopeCache` kernels dominate
- **Used for**: DeepSeek V2/V3 (also reused as preprocessing for DSA)
- **Source**: `vllm_ascend/attention/mla_v1.py` `AscendMLAImpl`

## KV cache compression

### `hamming_sparse` (KVComp)

- **vLLM-Ascend key**: `additional_config.hamming_sparse.enabled`
- **Default**: `False`
- **When True → profiling fingerprint**: `NpuHammingDistTopK` and `NpuSignBitsPack` kernels appear
- **Performance note**: if these kernels account for > 15% of attention block time (`block_class_summary.csv` for attention blocks), the compression overhead may outweigh memory savings
- **Agent note**: if detected as enabled but user didn't mention it, confirm this is intentional

## MoE dispatch fusion

### `enable_fused_mc2` (Fused MC2)

- **vLLM-Ascend key**: `additional_config.enable_fused_mc2` or env `VLLM_ASCEND_ENABLE_FUSED_MC2`
- **Default**: `0` (disabled)
- **Modes**:
  - `0`: standard alltoall + MC2 (no fusion)
  - `1`: fused `DispatchFFNCombine` for prefill (W8A8 only, EP ≤ 32, no MTP, no dynamic EPLB)
  - `2`: fused `DispatchGmmCombineDecode` for decode (W8A8 only, MTP must also be W8A8)

| Mode | Profiling fingerprint |
|------|----------------------|
| 0 (unfused, default) | Separate `MoeDistributeDispatchV2` + `MoeDistributeCombineV2` before/after alltoallv |
| 1 (prefill fused) | `DispatchFFNCombine` replaces dispatch + combine |
| 2 (decode fused) | `DispatchGmmCombineDecode` replaces dispatch + combine |

- **Agent note**: `config_signatures.moe_dispatch` reports fused vs unfused from kernel names. If unfused, check: is W8A8 quant active? Is EP ≤ 32? Is MTP active? These are prerequisites for fusion.

## Speculative decode

- **vLLM key**: `speculative_config`
- **Profiling fingerprint when active**: steps show `step_type = "speculative"` (from `speculative_layer_count > 0` in `step_summary.csv`)
- **Interaction with graph mode**: MTP requires graph mode for the draft model. The decode step's `cudagraph_capture_sizes` must cover `num_spec_tokens + 1`.
- **Agent note**: if `step_type = "speculative"` is detected but `graph_mode = "partial_capture"`, the most likely cause is `cudagraph_capture_sizes` not covering the speculative decode batch size.

## Parallelism

### Tensor parallelism (TP)

- **vLLM key**: `parallel_config.tensor_parallel_size`
- **Profiling fingerprint**: allreduce or reducescatter appears in attention/FFN blocks. `hccl_op_summary.csv` shows these collective types.
- **TP = 1 → fingerprint**: No HCCL collectives in compute blocks (only potential send/recv for KV transfer or barrier ops).

### Expert parallelism (EP)

- **vLLM key**: `parallel_config.enable_expert_parallel` → `expert_parallel_size`
- **Profiling fingerprint**: alltoallv appears before/after MoE expert blocks. `block_class_summary.csv` shows `comm_share_mean > 0` for `block_kind=moe`.

### TP + EP combined inference

- `config_signatures.parallelism` reports detected TP/EP from collective patterns.
- **Agent note**: if `TP × EP < rank_count`, there are likely additional DP or PP groups. Ask the user for their intended `tensor_parallel_size`, `expert_parallel_size`, `data_parallel_size`, `pipeline_parallel_size`.

### Pipeline parallelism (PP)

- **Profiling fingerprint**: Some ranks show `has_attention = False` in `rank_summary.csv`. These ranks have significantly fewer events and lower total wall time.
- `config_signatures.reduced_work_ranks` reports this.

### Fine-grained TP

- **vLLM-Ascend key**: `additional_config.finegrained_tp_config`
- **Allows**: different TP sizes per module (oproj, lmhead, embedding, mlp, olora)
- **Constraint**: `oproj_tensor_parallel_size` and `olora_tensor_parallel_size` only work in graph mode + PD decode node
- **Agent note**: not detectable from profiling data. Ask the user if they use fine-grained TP when TP overhead varies significantly across different modules.

## Communication optimization

### MatMul + AllReduce fusion

- **vLLM-Ascend key**: `VLLM_ASCEND_ENABLE_MATMUL_ALLREDUCE` (env) or `additional_config.enable_matmul_allreduce`
- **Default**: disabled (`0`)
- **When True → profiling fingerprint**: allreduce starts before MatMul ends (overlap). MatMul wall time is shorter, no gap before allreduce.
- **A2 only**

### FlashComm1

- **vLLM-Ascend key**: `VLLM_ASCEND_ENABLE_FLASHCOMM1` (env)
- **Default**: disabled (`0`)
- **Effect**: Optimized TP communication path. Better at high concurrency.

### FlashComm2

- **vLLM-Ascend key**: `VLLM_ASCEND_FLASHCOMM2_PARALLEL_SIZE` (env)
- **Default**: `0` (disabled). Set to N ≥ 1 to enable with O-matrix TP group size N.

## Other configurable features

### MLAPO (MLA prefill optimisation)

- **Key**: `VLLM_ASCEND_ENABLE_MLAPO` (env)
- **Default**: enabled (`1`)
- **Effect**: DeepSeek W8A8 performance optimization. Trades NPU memory for speed.
- **Agent note**: only relevant for MLA-backend models (DeepSeek V2/V3). If enabled but user reports OOM, suggest disabling it.

### Weight NZ (fractal layout)

- **Key**: `VLLM_ASCEND_ENABLE_NZ` (env)
- **Default**: `1` (quant only)
- **Modes**: 0 = disabled, 1 = quant only, 2 = BF16/FP16 also
- **Agent note**: not directly visible in profiling data. Only ask about this if the user reports weight-loading or memory-layout issues.

### Continuous batching (chunked prefill)

- **vLLM key**: `scheduler_config.enable_chunked_prefill`
- **Profiling fingerprint**: mix of prefill (`main_layer_count > 1`) and decode (`main_layer_count == 1`) steps in the same window
- **Agent note**: `config_signatures` has no dedicated field for this — check if both `step_type = "prefill"` and `step_type = "decode"` appear in `step_summary.csv`

### Dynamic EPLB

- **vLLM-Ascend key**: `additional_config.eplb_config.dynamic_eplb` + env `DYNAMIC_EPLB=true`
- **Default**: `False`
- **Profiling fingerprint**: Over long windows, `hccl_op_summary.csv` rank_skew_ratio for alltoallv should decrease as EPLB converges.

### Multi-stream DSA overlap

- **vLLM-Ascend key**: `additional_config.multistream_dsv4_dsa_overlap`
- **Default**: `True`
- **When True → profiling fingerprint**: attention compute time overlaps with HCCL communication in the DSA backend. Allreduce appears to run concurrently with attention — `aicore_time` and communication time overlap.

## Agent usage pattern

1. Read `characterizations.json` → `config_signatures` for detected states.
2. For each `detected` value that is not `unknown` or `not_applicable`, find the matching entry in this file.
3. Compare the detected state with the documented default. If they differ (or if the state indicates a suboptimal configuration), formulate a specific follow-up question referencing the exact config key.
4. Ask **only** the questions relevant to what was detected. Do not dump the entire config checklist.
5. If the user provides their vLLM-Ascend version, compare with the version header above. Warn if there is a major version gap.
6. If the user's answer reveals the config is intentional (e.g. `enforce_eager=True` for debugging), acknowledge this and move on. Do not argue.
