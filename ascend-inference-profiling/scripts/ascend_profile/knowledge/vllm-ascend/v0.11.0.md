## Graph compilation & execution mode

### `enforce_eager`

- **vLLM key**: `model_config.enforce_eager`
- **Default**: `False` (graph mode by default)
- **When `True` (eager) → profiling fingerprint**:
  - `step_anatomy.head_wall_ms / step_summary.wall_ms > 10%`
  - Many small kernels with high `wait_us`, gaps between sequential ops
- **When `False` (graph) → profiling fingerprint**:
  - `head_wall_ms / wall_ms < 5%`
  - Large fused kernel blocks, no inter-kernel gaps
- **Interaction**: vLLM-Ascend graph mode uses `enable_npugraph_ex`
  for compilation. Without it, graph mode falls back to torchair.
- **Source**: `vllm_ascend/compilation/compiler_interface.py` `AscendCompiler`
- **Agent note**: if `config_signatures.graph_mode = "eager_mode"`, ask whether
  `enforce_eager` was explicitly set to True (default is False).

### `enable_npugraph_ex`

- **vLLM-Ascend key**: `ascend_compilation_config.enable_npugraph_ex`
- **Default**: **not present** (v0.11.0 has no `AscendCompilationConfig`)
- **Source**: `vllm_ascend/ascend_config.py` `AscendCompilationConfig`

### `enable_static_kernel`

- **vLLM-Ascend key**: `ascend_compilation_config.enable_static_kernel`
- **Default**: **not present** (v0.11.0 has no `AscendCompilationConfig`)
- **Requires**: `enable_npugraph_ex=True`

### `cudagraph_capture_sizes`

- **vLLM key**: `compilation_config.cudagraph_capture_sizes`
- **Default**: vLLM dynamic default
- **Profiling fingerprint when incomplete**:
  - Mixed pattern: some steps graph-like, others eager-like
  - `config_signatures.graph_mode = "partial_capture"`
- **Agent note**: if partial_capture detected, ask for cudagraph_capture_sizes
  and num_speculative_tokens. Must cover num_spec_tokens + 1 for decode.

## Attention backends

vLLM-Ascend selects the attention backend automatically based on model architecture and quant config. The detected backend appears in `config_signatures.attention_backend`.

### FIA (dense models)

- **Detected when**: `FusedInferAttentionScore[V2-V5]` is the dominant attention kernel
- **Used for**: Qwen, Llama, Mistral, GLM, and other dense models
- **Architecture resolution**: `gqa_or_mha` — may refine to `mha`/`gqa`/`mqa` via shape-based heuristic in `attention_families.yaml`
- **Source**: `vllm_ascend/attention/attention_v1.py` `AscendAttentionBackend`

### SFA (sparse attention)

- **Detected when**: `QuantLightningIndexer`, `KVQuantSparseAttnSharedKV`, `Compressor` kernels appear
- **Used for**: DeepSeek V3.2 (DSA), DeepSeek V4 (CSA, HCA)
- **Sub-type resolution**: Determined by kernel signature combination (see `attention_families.yaml`)
- **Source**: `vllm_ascend/attention/sfa_v1.py` `AscendSFABackend`

### MLA (multi-head latent attention)

- **Detected when**: `MlaPreprocess`/`MlaProlog`/`KvRmsNormRopeCache` kernels dominate
- **Used for**: DeepSeek V2/V3 (also reused as preprocessing for DSA)
- **Source**: `vllm_ascend/attention/mla_v1.py` `AscendMLAImpl`

## Parallelism

### Tensor parallelism (TP)
- **vLLM key**: `parallel_config.tensor_parallel_size`
- **Profiling fingerprint**: allreduce/reducescatter in attention/FFN blocks
- **TP = 1 fingerprint**: No HCCL collectives in compute blocks

### Expert parallelism (EP)
- **vLLM key**: `parallel_config.enable_expert_parallel`
- **Profiling fingerprint**: alltoallv before/after MoE expert blocks

### Pipeline parallelism (PP)
- **Profiling fingerprint**: Some ranks show `has_attention = False`

### Fine-grained TP
- **vLLM-Ascend key**: `additional_config.finegrained_tp_config`
- **Agent note**: not detectable from profiling. Ask if TP overhead varies by module.

## Agent usage pattern

1. Read `characterizations.json` → `config_signatures` for detected states.
2. For each `detected` value that is not `unknown` or `not_applicable`, find the matching entry in this file.
3. Compare the detected state with the documented default. If they differ (or if the state indicates a suboptimal configuration), formulate a specific follow-up question referencing the exact config key.
4. Ask **only** the questions relevant to what was detected. Do not dump the entire config checklist.
5. If the user provides their vLLM-Ascend version, compare with the version header above. Warn if there is a major version gap.
6. If the user's answer reveals the config is intentional (e.g. `enforce_eager=True` for debugging), acknowledge this and move on. Do not argue.
