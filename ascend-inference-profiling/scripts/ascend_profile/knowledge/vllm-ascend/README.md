# vLLM-Ascend Knowledge Base

Three-layer structure for vLLM-Ascend performance knowledge.

## Layer 1: Diagnostic Playbook (shared, version-stable)

`diagnostic-playbook.md` — profiling signal → diagnosis → action. Auto-includes
version annotations *(vX.Y.Z+)* so the agent gets version-awareness without
cross-referencing. ~420 lines, grows slowly.

## Layer 2: Per-Version Config Guides (varying defaults)

`v{version}.md` — config defaults, profiling fingerprints, and agent notes for one
upstream version. Auto-generated by `generate_config_guide.py`, then reviewed.
Each file ~200 lines.

## Layer 3: Changelog (profiling-relevant changes)

`changelog.md` — ALL profiling-relevant changes in a single file. One read, all
version history. The agent scans `## v{version}` headings.

Entries per version (newest first): v0.22.1rc1, v0.21.0rc1, v0.20.2rc1, v0.18.0, v0.17.0rc1, v0.13.0, v0.12.0rc1, v0.11.0

## Version matching

1. Agent asks: "Which vLLM-Ascend version are you running?"
2. Agent loads the **exact match** config guide if available
3. If no exact match → **closest lower version** + warn the user
4. Agent uses the **diagnostic playbook** for signal-to-action (version annotations are inline)
5. Agent checks `changelog.md` only when the user's version falls between two known config guides
